## Findings

### F1. Cross-capsule attribution drift in `09ccc2eb` goal text

`capsule:09ccc2eb` (lint quality gate, ownerAgent=`dev`) carries the goal text:

> "Pricing/page.tsx errors handled inside capsule **252e26f1 by dev-2**."

But `capsule:252e26f1` is actually owned by `dev-3` (see `wanman capsule get
252e26f1`). The cross-reference is correct (the carved-out path *is* in
252e26f1's allowedPaths), but the agent attribution is wrong. A reader trying
to locate the responsible engineer to ask follow-up questions will route to the
wrong agent.

This is small but it is the second time we've seen capsule goals reference
peer capsules with hand-typed metadata that drifts from the real record. The
generalisable lesson is: prose like "owned by dev-X" inside a capsule goal will
silently rot whenever the peer capsule is reassigned. Either omit the
attribution or anchor it on the immutable capsule id only.

Skill emitted: `capsule-cross-reference-hygiene` (see "Skills" section).

### F2. Task `710ec182` acceptance is split across two capsules with no recorded dependency

`task:710ec182` says: "Acceptance: `npm run lint` src errors drop from 7 to 0."
Two capsules now together cover that scope:

- `capsule:09ccc2eb` (dev) — fixes 5 of 7 errors; allowedPaths excludes
  `pricing/page.tsx`. Its acceptance is rewritten to "at most 2 src errors
  remain."
- `capsule:252e26f1` (dev-3) — fixes the other 2 errors as a side effect of
  the BYO-key FAQ rewrite, but its parent task is `1cff6b5c` (BYO-key
  contradiction), not `710ec182`.

`capsule:09ccc2eb.blockedBy` is `[]`, but the task it serves cannot reach
"7 → 0" until `252e26f1` also merges. If `252e26f1` slips, the lint
quality-gate task remains effectively open even after its own capsule is
merged. The wanman model can express this — `blockedBy` accepts capsule ids —
we just didn't use it.

Recommendation (CEO-actionable): set
`capsule:09ccc2eb.blockedBy = [252e26f1]` on the task-completion side, OR
mark `task:710ec182` as `blockedBy [1cff6b5c]` so a reader knows the task is
not closeable on its own capsule alone.
